## 프로젝트 개요

롯데호텔 시그니엘 및 글로벌 체인(30개) 채널 재구축 및 예약/회원 고도화 및 디자인 리뉴얼 프로젝트입니다.
화이트 페이지 이슈 해결과 페이지 전환 속도 개선을 최우선 목표로 진행되었습니다.

프로젝트 아키텍처 설계, 애플리케이션 빌드/배포, 공통 모듈 및 컴포넌트 설계, 모바일 앱·웹뷰 브릿지 함수 설계,
개발표준 정의서 및 컴포넌트 명세서 작성/교육 등을 담당했습니다.

## 1. CSR/SPA의 초기 렌더링 최적화

### CSR/SPA 변환과 그 한계점

React.js 로의 리뉴얼을 통해 화면 전환 간 속도는 급격히 개선되었습니다. 그러나 글로벌 30개 체인의
방대한 콘텐츠를 다루는 CSR/SPA 환경에서 번들 사이즈 증가는 초기 렌더링 속도 저하의 주요원인이었습니다.

리뉴얼 프로젝트의 핵심목표인 성능 개선을 달성하기 위해 **초기 렌더링 시간 3초 이내 진입**과
**화면 전환시 화이트 페이지 제거**를 필수 과제로 설정했습니다.

### 효율적 코드 스플리팅을 위한 라우트 구조 설계 및 관리

<HighlightBox type="info" title="핵심 구현 사항">
- **IA 기반의 디렉토리 구조화**: IA와 URI 설계서를 기반으로 최상위 레벨을 체인명이 동적으로 적용되는 체인, 글로벌, 아라이 이 3가지로 분류했습니다.
- **RouteObject 기반의 선언적 관리**: Route 컴포넌트 대신 RouteObject를 사용하여 라우트 설정과 View를 분리했습니다.
- **DX를 고려한 경로 유틸리티 사용**: Vite의 동적 import 사용 시 상대경로만 허용되는 제약을 해결하기 위해 헬퍼 함수를 개발했습니다.
</HighlightBox>

### UX 향상을 위한 페이지 preloading 구현

페이지 lazy loading으로 인한, 화면 전환 시 발생하는 화이트 페이지 현상을 방지하고자
Next.js의 prefetching 기능을 벤치마킹한 프리로딩 기능을 구현했습니다.

- **Custom Link & Hook 구현**: 사용자가 링크에 호버(hover) 시 해당 경로의 페이지 청크를 미리 로드하는 Custom Link 컴포넌트와, 특정 로직 흐름에서 페이지 청크를 로드할 수 있는 usePreload 훅을 개발했습니다.
- **재귀적 라우트 탐색**: 라우트 객체를 재귀적으로 탐색하여 타겟 페이지 컴포넌트를 로드하는 방식을 적용했습니다.

## 2. 검색엔진 최적화(SEO)와 prerendering

### Prerendering 을 선택하게 된 이유

SEO 팀의 요구대로 당장 Next.js로 전환하기에는 다음과 같은 현실적 제약이 따랐습니다.

1. **인프라 오버헤드**: 당시 시스템 아키텍처는 AWS S3를 활용하여 프론트엔드를 정적호스팅 하도록 설계되어 있었습니다.
2. **러닝 커브와 일정**: React 경험이 적은 개발자들이 즉시 Next.js 생태계를 학습하며 개발하기에는 정해진 일정이 촉박했습니다.

이에 저는 기존 S3 인프라와 React 개발 환경을 유지하면서 SEO 요구사항을 충족할 수 있는 **절충안으로 prerendering**을 채택했습니다.

### Prerendering 프로세스 성능 개선: 병렬처리 도입

하지만 문제는 **약 10,000개에 달하는 페이지**를 처리하는 과정에서 심각한 병목 현상이 발생한다는 것이었습니다.

<HighlightBox type="warning" title="성능 이슈">
- **속도 저하**: 단일 스레드로 순차 처리 시, CMS 콘텐츠 누락이나 렌더링 지연에 따른 타임아웃 대기 시간이 누적되었습니다.
- **빌드 실패**: 전체 실행 시간이 Jenkins 파이프라인의 임계 시간(9시간)을 초과하여 정적 HTML 배포가 실패하는 경우가 발생했습니다.
</HighlightBox>

이를 해결하기 위해 안정적인 병렬 처리를 위해 커뮤니티 신뢰도가 준수한 **puppeteer-cluster를 도입**했습니다.

## 3. 아키텍처에 대한 고민과 FSD 적용

### 프로젝트 규모가 커질수록 드러나는 기존 아키텍처의 한계

기존 프론트엔드 프로젝트는 `src/` 하위에 views, components, hooks, api, router 등을 두는 역할 기반 디렉토리 구조를 사용했습니다.
소규모 프로젝트에서는 무리가 없었지만, **프로젝트 규모가 커질수록** 다음과 같은 한계가 명확해졌습니다.

1. **파일 탐색 비용의 증가**: 하나의 디렉토리 안에 수십 개의 파일이 누적되면서 원하는 파일과 코드를 찾는데 상당한 시간이 소요되었습니다.
2. **비즈니스 도메인 단위로의 응집도 부족**: 서로 다른 비즈니스 도메인의 코드가 동일한 디렉토리에 혼재되었습니다.
3. **역할과 책임의 모호성**: 전역 설정, 레이아웃, 재사용 모듈, 컴포넌트, 컴포즈 컴포넌트가 명확한 기준 없이 공존했습니다.

### 대안으로써 FSD(Feature-Sliced-Design)를 채택한 이유

FSD는 역할에 따라 **app → pages → widgets → features → entities → shared**로 이어지는 명확한 계층 구조를 가지며
각 계층 내부에서는 slice 단위로 비즈니스 도메인을 묶고, 그 안에서 다시 ui, api, lib 등 기능에 따라 segment를 세분화합니다.

<HighlightBox type="success" title="FSD 도입 효과">
- **의존성 감소와 응집도 향상**: 계층과 slice를 기준으로 모듈을 분리함으로써, UI 변경이 비즈니스 로직이나 API 계층에 전파되지 않았습니다.
- **역할과 책임이 명확한 협업 구조**: 공통 개발자는 app, shared 레이어에 집중하고, 각 담당자는 자신이 맡은 도메인의 pages부터 entities 레이어까지만 관리했습니다.
- **직관적인 코드 추적**: 계층 구조와 디렉토리명만 따라가도 모듈의 성격과 소속된 비즈니스 도메인을 파악할 수 있었습니다.
</HighlightBox>
