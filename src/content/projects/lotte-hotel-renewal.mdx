## 프로젝트 개요

롯데호텔 시그니엘 및 글로벌 체인(30개) 채널 재구축 및 예약/회원 고도화 및 디자인 리뉴얼 프로젝트입니다.
기존 시스템의 화이트 페이지 이슈 해결과 페이지 전환 속도 개선을 최우선 목표로 진행되었습니다.

프로젝트 아키텍처 설계, 애플리케이션 빌드/배포, 공통 모듈 및 컴포넌트 설계, 모바일 앱·웹뷰 브릿지 함수 설계,
개발표준 정의서 및 컴포넌트 명세서 작성/교육 등을 담당했습니다.

## 1. CSR/SPA의 초기 렌더링 최적화

### CSR/SPA 변환과 그 한계점

React 로의 리뉴얼을 통해 화면 전환 간 속도는 급격히 개선되었습니다. 그러나 글로벌 30개 체인의
방대한 콘텐츠를 다루는 CSR/SPA 환경에서 번들 사이즈 증가는 초기 렌더링 속도 저하의 주요원인이었습니다.

리뉴얼 프로젝트의 핵심목표인 성능 개선을 달성하기 위해 **초기 렌더링 시간 3초 이내 진입**과
**화면 전환시 화이트 페이지 제거**를 필수 과제로 설정했습니다.

### 효율적 코드 스플리팅을 위한 라우트 구조 설계 및 관리

수백 개의 라우트를 효과적으로 분리하고 개발 생산성을 높이기 위해, IA 와 URI
구조에 기반한 디렉토리 구조를 구축했습니다. 뿐만 아니라 효율적인 코드 스플리팅
관리를 위하여 구조화된 디렉토리 구조를 설계할 필요가 있었습니다.

<HighlightBox type="info" title="핵심 구현 사항">
  - **IA 기반의 디렉토리 구조화**: IA와 URI 설계서를 기반으로 최상위 레벨을
  체인명이 동적으로 적용되는 체인과 글로벌, 아라이 이 3가지로 분류했습니다. 하위
  디렉토리는 URI 체계를 기반으로 디렉토리를 생성했으며, 물리적 깊이를 통일하기
  위해 특정 깊이 이상은 `“_”` 를 활용해 연결하는 규칙을 적용했습니다. 이를 통해
  방대한 라우트 환경에서도 manualChunks 설정과 같이 번들링 최적화를 위한 파일
  구조의 일관성을 확보했습니다. <br />
  - **RouteObject 기반의 선언적 관리**: `<Route />` 컴포넌트 대신
  `RouteObject[]` 를 사용하여 라우트 설정과 View 를 분리했습니다. 이 또한 체인,
  글로벌, 아라이 이 3 가지 기준으로 디렉토리를 분기하고 그 아래에 IA 첫번째
  레벨로 파일을 생성해 독립적으로 라우트를 관리하게 했습니다. IA 구조를 따라
  개발자들에게 작업이 분배되었는데, 이를 활용하여 각자 책임지고 있는 영역의
  라우트를 명확히 담당함으로써 협업 시 코드 충돌을 방지할 수 있었습니다. 또한
  진입 전 데이터 prefetching 등의 기능을 구현하려면 `RouteObject[]` 구조를
  사용해야 했기에 이는 더욱 좋은 선택이었습니다. <br />
  - **DX를 고려한 경로 유틸리티 사용**: Vite 의 동적 import 사용 시
  상대경만 허용되는 제약을 해결하기 위해 pages 레이어부터 경로를 작성하면 올바른
  상대 경로로 반환하는 헬퍼 함수를 개발했습니다. 이를 통해 개발 편의성을 높이고
  경로 관리의 포인트를 하나의 함수로 일원화 했습니다.
</HighlightBox>

### UX 향상을 위한 페이지 preloading 구현

페이지 lazy loading으로 인한, 화면 전환 시 발생하는 화이트 페이지 현상을
방지하고자 Next.js의 prefetching 기능을 벤치마킹한 프리로딩 기능을
구현했습니다.

<HighlightBox type="info" title="핵심 구현 사항">
  - **Custom Link & Hook 구현**: 사용자가 링크에 호버(hover) 시 해당 경로의
  페이지 청크를 미리 로드하는 `Custom Link` 컴포넌트와, 특정 로직 흐름에서
  페이지 청크를 로드할 수 있는 `usePreload` 훅을 개발하여 팀 내 표준으로
  지정했습니다.<br />
  - **재귀적 라우트 탐색**: 라우트 객체를 재귀적으로 탐색하여 타겟 페이지
  컴포넌트를 로드하는 방식을 적용했습니다. 서비스의 평균 URL Depth 가 1-4 단계로
  얕은 점을 고려했을 때, 재귀의 성능 부담은 미미하면서도 코드의 가독성과
  유지보수성을 확보할 수 있었습니다.
</HighlightBox>

### 리소스 최적화 및 캐시버스팅

인프라팀과의 협업을 통해 캐싱을 효과적으로 적용시킬 수 있었습니다. 이를 통해 리소스 로딩 속도를 높이고 버전관리 이슈를 해결할 수 있었습니다. 또한 배포 직후 발생할 수 있는 에러를 방지했습니다.

<HighlightBox type="info" title="핵심 구현 사항">
- **인프라 협업과 빌드 시 번들 압축**: 인프라팀과 협업하여 CloudFront 의 캐싱 정책을 세웠으며 vite 빌드
  시 `terser`를 활용한 번들 파일 압축을 통해 리소스 크기를 최소화하고 캐싱을 활용할 수 있었습니다.
- **리소스 캐시 갱신 자동화**: vite 빌드 시 해시값이 붙는 번들 파일 외에도, 외부로부터 참조하는
  CSS/JS 파일의 캐시 문제를 해결하기 위해 `vite-plugin-html` 을 활용했습니다. 빌드 시점에
  타임스탬프를 리소스의 쿼리스트링 값으로 주입함으로써 배포 시 사용자가 최신 버전을 확인할 수
  있도록 했습니다. <br/>
- **배포감지 및 청크 로드 에러 처리**: 사용자가 서비스를 이용하는 도중 새 버전이 배포되어 해시값이
  변경될 경우 청크 로드 에러를 방지했습니다. vite 공식문서를 확인해보니 `“vite:preloadError”` 라는
  이벤트를 통하여 이전의 청크를 임포트를 시도할 때 해당 에러가 발생한다는 것을 확인했습니다. 이를
  활용하여 에러 감지 시 새로운 버전이 배포된 것으로 간주하여 페이지를 리로드함으로써 사용자가
  끊김 없이 최신 버전을 경험할 수 있도록 했습니다.
</HighlightBox>

## 2. 검색엔진 최적화(SEO)와 prerendering

### Prerendering 을 선택하게 된 이유

개발 기간 중 SEO 팀의 요구대로 당장 Next.js로 전환하기에는 다음과 같은 현실적 제약이 따랐습니다.

<HighlightBox type="error">
1. <strong className="text-error font-semibold">인프라 오버헤드</strong>: 당시 시스템 아키텍처는 AWS S3를 활용하여 프론트엔드를 정적호스팅 하도록 설계되어 있었습니다. 이를 Next.js등의 SSR 로 전환하려면 CDN을 AWS EKS 등으로 인프라를 전면 재구축해야
   했습니다. 이는 단순 서버 비용 증가를 넘어 배포 파이프라인 재설계 및 서버 상태 모니터링, 트래픽
   대응을 위한 운영 리소스 투입 등 기하급수적 비용증가를 의미했습니다.
2. <strong className="text-error font-semibold">러닝 커브와 일정</strong>: React 경험이 적은 개발자들이 즉시 Next.js 생태계를 학습하며 개발하기에는 정해진 일정이 촉박했습니다.
</HighlightBox>

이에 저는 기존 S3 인프라와 React 개발 환경을 유지하면서 SEO 요구사항을 충족할 수 있는 **절충안으로 prerendering**을 채택했습니다.

동적 URL 을 포함**하여 생성한 <strong className="font-semibold text-secondary">sitemap.xml 을 기반으로 정적 HTML 을 갱신</strong>하는 순환 구조를

    - <strong className="font-semibold">Await</strong>: Puppeteer 가 완성된 콘텐츠를 크롤링할 수 있도록 했습니다. Cms 데이터 페칭과

정의하고 이를 구현함으로써 **리스코프 치환의 원칙**을 준수하고자 했습니다.

### Prerendering 프로세스 성능 개선: 병렬처리 도입

하지만 문제는 **약 10,000개에 달하는 페이지**를 처리하는 과정에서 <strong className="text-warning font-semibold">심각한 병목 현상</strong>이 발생한다는 것이었습니다.

<HighlightBox type="warning" title="성능 이슈">
  - <strong className="text-warning font-semibold">속도 저하</strong>: 단일 스레드로 순차 처리 시, CMS 콘텐츠 누락이나 렌더링 지연에 따른 타임아웃
대기 시간이 누적되었습니다. 타임아웃 시간을 기본값보다 짧게 했음에도 <strong className="text-warning font-semibold">1,000 페이지당 약
4 시간</strong>이 소요되었습니다.<br />

- <strong className="text-warning font-semibold">빌드 실패</strong>: 전체 실행
  시간이 Jenkins 파이프라인의 임계 시간(9시간)을 초과하여 정적 HTML 배포가
  실패하는 경우가 발생했습니다.
</HighlightBox>

이를 해결하기 위해 **안정적인 병렬 처리**를 위해 커뮤니티 신뢰도가 준수한<span className="text-subtle">(weekly download 6 만+, github star 3.5K)</span> **puppeteer-cluster** 를 도입했습니다.

<HighlightBox type="info" title="핵심 구현 사항">
  - **동시성 제어**: 시스템 안정성을 위해 maxConcurrency 설정을 전체 코어의 75% 수준에서
워커(worker)를 운용 하도록 하였습니다. 단순히 많이 사용하기만 하면 Out of Memory 로
프로세스가 죽는 현상이 발생했습니다. 나머지 25% 여유 자원은 OS 의 기본 동작과 Node.js 의
이벤트 루프 처리, 가비지 컬렉션을 위해 남겨두어 시스템의 안정성을 보장하기 위함이었습니다.
결과적으로 이 비율이 제게 주어진 환경에서는 자원을 최대한 활용하면서도 완주할 수 있는 최적의
지점이었습니다.
  - **모니터링**: 각 워커에 ID 를 부여하고 각 작업에 난수 ID 를 부여하여 로그 추적 및 디버깅에 용이할
수 있도록 했습니다.
</HighlightBox>

결과적으로 <strong className="font-semibold text-secondary">1,000 페이지당 4 시간 이상 소요되던 작업을 30 분 내외로 단축</strong> 하는 성과와
메모리 오버플로우 없이 Jenkins 파이프라인 제한 시간 내 안정적인 배포에 성공할 수 있었습니다.

구현 단계에서 인프라 접근 제약으로 인해 CDN 레벨의 봇 분기 처리가 어려운 상황이었습니다. 이에 대한
대안으로 저는 클라이언트 진입점에서 User-Agent 를 식별하여 prerendering 된 정적 HTML 로 라우팅 하는

이 과정에서 봇이 접근하는 정적 HTML 경로와 실제 원본 URL 이 달라져 발생할 수 있는 중복 콘텐츠 문제를
해결하기 위해 모든 페이지마다 **Canonical Tag**(`<link rel=
”canonical”/>`)을 주입 하였습니다. 이렇게 함으로써
정적 HTML 파일 내부에 **원본 URL 을 가리키는 Canonical Tag 를 통해 검색엔진 평가가 원본 경로로
귀속**되도록 설계했습니다.


### 회고
LSP 를 적용하여 크롤링 대상 경로를 Route 기반으로 할지, sitemap 기반으로 할지 유연하게 변경할 수 있는
구조를 짠 점은 만족스럽지만, 크롤링 소스를 변경할 때마다 코드를 수정하고 배포해야 하는 점이
번거로웠습니다. CLI 환경변수를 통해 실행 시점에 전략을 주입 받도록 구성했다면 코드 수정 없이 운영
유연성을 더 높일 수 있었을 것이라는 아쉬움이 남습니다. 향후 이 부분을 개선하고 개발자 경험을 높이고 싶습니다.


## 3. 아키텍처에 대한 고민과 FSD 적용

### 프로젝트 규모가 커질수록 드러나는 기존 아키텍처의 한계

기존 프론트엔드 프로젝트는 `src/` 하위에 views, components, hooks, api, router 등을 두는 역할 기반 디렉토리 구조를 사용했습니다.
소규모 프로젝트에서는 무리가 없었지만, **프로젝트 규모가 커질수록** 다음과 같은 한계가 명확해졌습니다.

<HighlightBox type="warning">
1. <strong className="text-warning font-semibold">파일 탐색 비용의 증가</strong>: 하나의 디렉토리 안에 수십 개의 파일이 누적되면서 원하는 파일과 코드를 찾는데 상당한 시간이 소요되었습니다.
2. <strong className="text-warning font-semibold">비즈니스 도메인 단위로의 응집도 부족</strong>: 서로 다른 비즈니스 도메인의 코드가 동일한 디렉토리에 혼재되다 보니, 특정 기능을 이해하거나 수정하기
위해 관련 파일들을 추적하는 데에 많은 번거로움이 있었습니다.
3. <strong className="text-warning font-semibold">역할과 책임의 모호성</strong>: 전역 설정, 레이아웃, 재사용 모듈, 컴포넌트, 컴포즈 컴포넌트가 명확한 기준 없이 공존하면서 코드의 성격을
디렉토리 구조만으로 파악하기 어려웠습니다.
</HighlightBox>

이러한 구조는 소규모 프로젝트에서는 큰 문제가 되지 않지만, 장기적으로 유지, 확장해야 하는 대규모
프로젝트에는 적합하지 않다고 판단했습니다. 이에 따라, 사내 프론트엔드의 기준이 될 수 있는 아키텍처를
정립하고자 다음과 같은 원칙을 세웠습니다.

<HighlightBox type="info">
- **관심사의 분리**: 역할과 책임에 따른 명확한 계층 구조를 가질 것
- **도메인 중심 응집**: 특정 비즈니스 로직에 관련되어 있는 모듈은 물리적으로 인접해 있을 것
- **프레임워크 독립성**: 비즈니스 로직은 특정 라이브러리나 프레임워크에 과도하게 종속되지 않을 것
- **러닝커브 극복**: React 경험이 적은 백엔드 개발자의 비율이 높았는데 이들에게 쉽게 다가올 수 있는
구조일 것
</HighlightBox>


### 대안으로써 FSD(Feature-Sliced-Design)를 채택한 이유

FSD는 역할에 따라 **app → pages → widgets → features → entities → shared**로 이어지는 명확한 계층 구조를 가지며
각 계층 내부에서는 slice 단위로 비즈니스 도메인을 묶고, 그 안에서 다시 ui, api, lib 등 기능에 따라 segment를 세분화합니다.
이를 기본 아키텍처로 도입한 결과, 다음과 같은 효과를 얻을 수 있었습니다.

<HighlightBox type="success" title="FSD 도입 효과">
  - <strong className="font-semibold text-secondary">의존성 감소와 응집도 향상</strong>: 계층과 slice를 기준으로 모듈을 분리함으로써, UI 변경이 비즈니스 로직이나 API 계층에 전파되지 않았습니다.
  이런 이유로 요구사항 변경 시 영향 범위를 예측 가능하게 관리할 수 있었습니다.
  - <strong className="font-semibold text-secondary">역할과 책임이 명확한 협업 구조</strong>: 공통 개발자는 app, shared 레이어에 집중하고, 각 담당자는
  자신이 맡은 도메인의 pages부터 entities 레이어까지만 관리하면 되도록 책임 범위를 명확히 나눌 수 있었습니다.
  - <strong className="font-semibold text-secondary">직관적인 코드 추적</strong>: 계층 구조와 디렉토리명만 따라가도 모듈의 성격과 소속된
  비즈니스 도메인을 파악할 수 있으므로, 파일명을 정확히 기억하지 못하더라도 탐색 비용이 크게 줄었습니다.
</HighlightBox>


### FSD 적용의 단점과 현실적인 제약
하지만 프로젝트를 진행하며 <strong className="font-semibold text-warning">FSD 적용의 단점과 현실적인 제약</strong>도 함께 경험했습니다.
